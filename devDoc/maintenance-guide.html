改修基本フロー

要件・影響範囲を確認する

何を変えるか、どのファイルが関係するかを洗い出す。本手順書の「影響範囲の調べ方」を参照。

開発環境でコードを修正する

ブランチを切ってから作業する: git checkout -b feature/xxxx

ローカルで動作確認する

docker compose up -d --build で確認。テスト仕様書のシナリオテストも実施。

セキュリティテストを実施する

権限チェック・allowlist・入力バリデーションが正しく動作しているか確認する。

本番環境へデプロイする

DBバックアップを取得してから実施。デプロイ後に動作確認。

ドキュメントを更新する

devDoc 内の関連ドキュメントを変更内容に合わせて更新する。

テーブル追加手順

例: 「VENDORS（取引先管理）」テーブルを新規追加する場合
schema.sql にテーブルを追加 
CREATE TABLE IF NOT EXISTS vendors (
    vendor_id   SERIAL PRIMARY KEY,
    vendor_name VARCHAR(255),
    contact     VARCHAR(255),
    status      VARCHAR(50),
    remarks     TEXT
);
allowlist に追加（3ファイル） 
// TableController.java
private static final Set<String> ALLOWED_TABLES = Set.of(
    "PROJECTS", "REQUIREMENTS", "SPECIFICATIONS",
    "APPLICATIONS", "ENVIRONMENTS",
    "VENDORS"  // ← 追加
);

// CsvImportController.java（同様に追加）
// CsvBulkController.java（同様に追加）
フロントエンドのラベル定義を追加 
// frontend/utils/labels.js
export const TABLE_LABELS = {
  PROJECTS: "案件管理",
  // ...
  VENDORS: "取引先管理",  // ← 追加
};

export const COLUMN_LABELS = {
  // ...
  VENDOR_ID: "取引先ID",
  VENDOR_NAME: "取引先名",
  CONTACT: "担当者",
};
動作確認

docker compose up -d --build → テーブル一覧にVENDORSが表示されること / CRUD操作が可能なこと / CSV出力対象に含まれること

本番DBにテーブルを作成 
# 本番ではschema.sqlは自動実行されないため手動でDDLを実行
docker compose exec db psql -U $DB_USERNAME -d dbmanagement \
  -c "CREATE TABLE IF NOT EXISTS vendors (...);"

カラム追加手順

例: PROJECTSテーブルに「budget（予算）」カラムを追加する場合
schema.sql を更新

CREATE TABLE IF NOT EXISTS に追記（開発環境の初期化用）。本番は別途ALTER TABLEで対応。

-- schema.sql の projects テーブルに追記
budget NUMERIC(15, 0),

-- 本番環境への反映コマンド（手動実行）
ALTER TABLE projects ADD COLUMN IF NOT EXISTS budget NUMERIC(15, 0);
フロントエンドのラベル・フィールド定義を追加 
// frontend/utils/labels.js
export const COLUMN_LABELS = {
  // ...
  BUDGET: "予算",
};

export const FIELD_DEFS = {
  // 入力タイプを定義（text / date / select）
  BUDGET: { type: "text" },
};
動作確認

一覧にカラムが表示されること / 追加・編集フォームに項目が表示されること / CSV出力に含まれること

⚠ 本番DBへのカラム追加: schema.sql は本番で自動実行されない（init.mode=never）。 必ず手動で ALTER TABLE を本番DBに実行してからアプリをデプロイすること。 カラム追加の前に必ずバックアップを取得すること。

APIエンドポイント追加手順

コントローラにメソッドを追加 
// 既存コントローラへの追加、または新規コントローラ作成
@GetMapping("/api/xxxx")
public ResponseEntity<?> newEndpoint() {
    // 権限チェックを必ず実装
    if ("VIEWER".equals(SecurityUtils.getCurrentRole())) {
        return ResponseEntity.status(403).body(Map.of("error", "権限がありません"));
    }
    // ... 処理
}
SecurityConfig の確認

新しいパスが /api/** に含まれていれば追加設定不要。ログイン不要なパスを追加する場合は SecurityConfig.java の requestMatchers に追記する。

フロントエンドのAPI呼び出しを追加 
// 対象 View の <script> セクション
import api from '@/api/index.js';

const result = await api.get('/api/xxxx');
// または
const result = await api.post('/api/xxxx', requestBody);
テスト仕様書にテストケースを追加

devDoc/10-test-spec.html の結合テスト・シナリオテストに追記する。

本番デプロイ手順

DBバックアップを取得 
docker compose exec db pg_dump -U $DB_USERNAME dbmanagement \
  > backup_before_deploy_$(date +%Y%m%d_%H%M%S).sql
DBスキーマ変更がある場合は先に実行 
# テーブル追加・カラム追加のDDLを手動実行
docker compose exec db psql -U $DB_USERNAME -d dbmanagement \
  -f /path/to/migration.sql
コードを取得・ビルド・再起動 
git pull origin main
docker compose up -d --build
動作確認 
docker compose ps
docker compose logs app --tail=50
# ブラウザでアクセスして主要機能を確認
異常時のロールバック 
# 旧バージョンに戻す
git checkout [前のコミットハッシュ]
docker compose up -d --build

# DBも必要な場合はバックアップからリストア

改修チェックリスト

コーディング時

  • 新しいAPIエンドポイントに権限チェック（VIEWER / ADMIN）が実装されているか
  • テーブル名・カラム名の動的SQLは allowlist または固定値のみを使用しているか（SQLインジェクション防止）
  • エラーメッセージに内部情報（SQL・スタックトレース等）を含んでいないか
  • 新しいデータ操作に操作履歴記録（OperationHistoryService.log()）が追加されているか
  • ファイル入出力でトランザクション（@Transactional）が必要な箇所に付いているか
  • 秘密情報（パスワード・APIキー等）をコードやGitにコミットしていないか

テスト時

  • 正常系だけでなく異常系（権限不足・不正パラメータ）のテストをしたか
  • ADMIN / EDITOR / VIEWER の3ロールすべてで動作確認したか
  • 既存機能への影響（デグレード）がないか確認したか
  • CSVインポートのトランザクション（失敗時ロールバック）を確認したか（変更した場合）

デプロイ前

  • DBバックアップを取得したか
  • DBスキーマ変更がある場合、本番DBへの手動DDL実行を準備したか
  • 関連するドキュメント（devDoc）の更新を行ったか
  • ロールバック手順を確認したか

デプロイ後

  • docker compose ps でコンテナが正常起動しているか
  • アプリログにエラーが出ていないか（docker compose logs app）
  • ブラウザで主要機能が動作するか確認したか
  • 操作履歴テーブルに正しく記録されているか
